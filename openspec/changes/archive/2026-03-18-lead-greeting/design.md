## Context

The repo-lead agent is dispatched via `harness lead --repo .` which spawns an interactive Claude Code session with a system prompt (the lead persona from `.harness/agents/lead.md`) and appended context (gathered repo state). The greeting behavior is entirely driven by the persona prompt — the LLM reads the instructions and generates a greeting based on the repo context injected via `--append-system-prompt`.

Currently, the Interactive Mode section of the persona says "introduce yourself with awareness of the specific repo" and lists 4 steps (name repo, summarize state, suggest directions, wait). It works but doesn't explain what the lead *can do* — users who don't already know the harness workflow miss most of its capabilities.

## Goals / Non-Goals

**Goals:**
- The lead's greeting clearly communicates its role as the human's interface to the harness pipeline
- The greeting surfaces concrete capabilities (implement issues, dispatch changes, explore ideas, create proposals, answer questions, analyze the repo)
- Suggested tasks are grounded in the repo's actual state (ready changes, open issues, assessment gaps) rather than generic
- The persona prompt stays concise enough that it doesn't bloat the system prompt or confuse the agent

**Non-Goals:**
- No code changes to `lead.py` or the CLI — the greeting is LLM-generated from the persona prompt
- No changes to the context gathering logic — the data is already there, the persona just needs to use it better
- No dynamic greeting templates or code-generated greeting text — keep it as natural language instructions in the persona

## Decisions

### Persona prompt structure over code-generated greetings

The greeting will be defined entirely in the persona prompt markdown (`.harness/agents/lead.md`), not generated by code. The lead agent is a Claude Code session — it reads its persona, reads the appended context, and generates the greeting.

**Alternative considered:** Building the greeting in `lead.py` and passing it as part of the prompt. Rejected because it couples greeting content to Python code, makes iteration slow, and fights the natural strength of having an LLM craft contextual responses.

### Capability categories organized by user intent

Present capabilities grouped by what the user wants to accomplish, not by harness internals:

1. **Build something** — implement GitHub issues, dispatch OpenSpec changes
2. **Plan something** — explore ideas, create proposals, design features
3. **Understand something** — answer repo questions, analyze code, review assessment scores

This maps to how users think ("I want to fix issue #42") rather than how the harness works ("dispatch a worker with --issue flag").

### Greeting structure: role → state → suggestions

Keep the existing 4-step structure but enhance it:
1. **Role statement** — one sentence explaining what the lead is and how it fits in the harness
2. **Current state** — pulled from context (ready changes, recent runs, assessment scores)
3. **Capability-aware suggestions** — 2-3 concrete suggestions that demonstrate different capabilities
4. **Open prompt** — wait for direction

The suggestions should rotate across capability categories so users see breadth, not just "here are the ready changes."

## Risks / Trade-offs

- **Prompt length vs. clarity** — Adding capability descriptions makes the persona longer. Mitigation: keep descriptions to one line each, use the example greeting to demonstrate rather than over-explain.
- **Stale example** — The example greeting in the persona will drift from reality as capabilities change. Mitigation: keep the example focused on structure/tone, not specific feature names.
- **Over-promising** — Listing capabilities the lead can't actually deliver well (e.g., "perform deep code analysis") could frustrate users. Mitigation: tie capabilities to actual harness commands and skills the lead has access to.
